# Requirements - Approval Flow System

**Version:** 1.0
**Last Updated:** 2026-01-30
**Project:** Approval Flow System
**Status:** ✅ COMPLETE - All Requirements Validated

## v1 Requirements

### Authentication & User Management

#### AUTH-01: User Authentication and Role Assignment ✅
**User Story:** As a system administrator, I want users to authenticate with role-based access control so that only authorized users can access to system and perform appropriate actions.

**Acceptance Criteria:**
- [x] User can log in with email/password or social login
- [x] System assigns one of three roles: Admin, General Department User, Engineering User
- [x] Session persists across browser refreshes
- [x] User can log out from any page
- [x] Unauthorized users cannot access protected pages

**Outcome:** Validated - Clerk authentication working with role-based access control

**Notes:** Use Clerk for modern authentication with RBAC. No external user access in v1.

#### AUTH-02: User and Department Management ✅
**User Story:** As an admin, I want to create and manage users and assign them to departments so that approval workflow knows where requests originate.

**Acceptance Criteria:**
- [x] Admin can create users with name, email, department assignment, and role
- [x] Admin can create departments with name and description
- [x] Admin can edit user details (name, email, department, role)
- [x] Admin can deactivate users (soft delete, preserves audit trail)
- [x] Admin can edit department details
- [x] Non-admins cannot access management screens

**Outcome:** Validated - Complete CRUD functionality for users and departments with edit capability added in Phase 8

**Dependencies:** AUTH-01

#### AUTH-03: Level-Based User Assignment for Approval Hierarchies ✅
**User Story:** As an admin, I want to assign users to levels within their department so that configurable approval hierarchies can route requests correctly.

**Acceptance Criteria:**
- [x] Admin can assign users to numeric levels within their department (Level 1, Level 2, Level 3, etc.)
- [x] Multiple users can be assigned to same level
- [x] Each user has exactly one level assignment per department
- [x] Approval routing uses level assignments to determine next approver
- [x] Admin can change user level assignments (updates active workflows immediately)

**Outcome:** Validated - Drag-and-drop hierarchy builder enables level assignment with immediate workflow effect

**Dependencies:** AUTH-02, CONF-03

### Request Management

#### REQ-01: Create Request with Form ✅
**User Story:** As a requester from a general department, I want to submit requests with title and description so that engineering can review and respond to my needs.

**Acceptance Criteria:**
- [x] User can create request with title (required) and description (required)
- [x] Form validates that required fields are filled
- [x] Request is created in "ImprovementRequest" status
- [x] Request is automatically associated with requester's department
- [x] Request creation timestamp is recorded for audit trail
- [x] Requester can view their created requests

**Outcome:** Validated - Request creation form with validation working

**Dependencies:** AUTH-01, AUTH-02

#### REQ-02: File Attachments for Requests ✅
**User Story:** As a requester, I want to attach supporting documents to my request so that engineering has full context for survey, design, and cost estimation.

**Acceptance Criteria:**
- [x] User can upload one or more files with request
- [x] Each file has a description field (optional)
- [x] System validates file types (PDF, images, Office docs)
- [x] System enforces size limits (configurable by admin)
- [x] Files are stored securely (local storage with public/uploads/)
- [x] Request detail page shows all attached files with download links
- [x] File uploads are logged in audit trail

**Outcome:** Validated - Local file storage implemented (user-approved switch from S3)

**Dependencies:** REQ-01

#### REQ-03: Request Status Tracking ✅
**User Story:** As a requester or approver, I want to see the current status of a request so that I know where it is in the workflow.

**Acceptance Criteria:**
- [x] Request displays one of these statuses: ImprovementRequest, SentToEngineer, DesignCostEstimationApproval, SendBackToRequester, Completed
- [x] Status change is timestamped in audit trail
- [x] Status determines which actions are available to which users
- [x] Current status is prominently displayed on request detail page
- [x] Status is included in list views and filters

**Outcome:** Validated - All status tracking working with StatusBadge component

**Dependencies:** REQ-01

### Approval Workflow Engine

#### WFL-01: Configurable Level-Based Approval Hierarchies ✅
**User Story:** As an admin, I want to configure approval hierarchies per department using levels so that business users can maintain workflows without code changes.

**Acceptance Criteria:**
- [x] Admin can create/edit approval hierarchy for each department
- [x] Hierarchy is organized by levels (Level 1, Level 2, Level 3, etc.)
- [x] Each level contains one or more users
- [x] Drag-and-drop UI to reorder levels and reassign users
- [x] Hierarchy configuration is versioned (changes don't break active workflows)
- [x] Admin can preview hierarchy before activating
- [x] Active requests follow hierarchy from time of creation (changes don't affect in-flight requests)

**Outcome:** Validated - Drag-and-drop hierarchy builder with batch save and validation

**Dependencies:** AUTH-03, CONF-03

#### WFL-02: Any-One-Per-Level Approval ✅
**User Story:** As an approver, I want any member of my level to approve a request so that bottlenecks are reduced and workflow doesn't wait for a specific person.

**Acceptance Criteria:**
- [x] When multiple users are at same level, any one can approve
- [x] Once one user at level N approves, request advances to level N+1
- [x] Remaining users at level N see request as "approved by peer" and cannot act
- [x] System tracks which user approved in audit trail
- [x] No approval required from all users at same level (any-one model)

**Outcome:** Validated - Any-one-per-level approval logic working

**Dependencies:** WFL-01

#### WFL-03: Sequential Approval Routing ✅
**User Story:** As a system, I want to route requests through approval levels sequentially so that all required approvals are gathered in correct order.

**Acceptance Criteria:**
- [x] Request routes through levels in configured order (Level 1 → Level 2 → Level 3, etc.)
- [x] Request waits at current level until approval from any user at that level
- [x] Once approved at level N, request automatically routes to level N+1
- [x] If requester is at top level, request routes directly to engineering
- [x] Routing logic is enforced at database level (no race conditions)
- [x] System logs each routing step in audit trail

**Outcome:** Validated - Sequential routing with createApprovalChain logic

**Dependencies:** WFL-01, WFL-02

#### WFL-04: Approve/Reject with Comments ✅
**User Story:** As an approver, I want to approve or reject requests with required comments so that decisions are documented and transparent.

**Acceptance Criteria:**
- [x] Approver can approve request (must enter comment)
- [x] Approver can reject request (must enter comment)
- [x] Comment field is required (cannot be empty)
- [x] Approval/rejection action is timestamped in audit trail
- [x] Rejected request returns to requester with "Rejected" status
- [x] Requester can view rejection reason and resubmit if needed
- [x] Approved request advances to next level/phase

**Outcome:** Validated - Approve/reject functionality with required comments

**Dependencies:** WFL-03

#### WFL-05: Engineering Solution Routing ✅
**User Story:** As a system, I want to route approved requests to engineering for solution development, then route solutions back through approvals for requester's department, so that full lifecycle is tracked.

**Acceptance Criteria:**
- [x] When request completes department approvals, status changes to "SentToEngineer"
- [x] Engineering user can submit solution with title, description, and file attachments
- [x] Solution submission changes status to "DesignCostEstimationApproval"
- [x] Solution routes through engineering department approval hierarchy
- [x] If engineering approver is top-level, solution goes directly to "SendBackToRequester"
- [x] Requester department can view solution when status is "SendBackToRequester"
- [x] Requester can download solution files and approved evidence

**Outcome:** Validated - Solution submission workflow with custom approval chains

**Dependencies:** WFL-03, WFL-04, REQ-02

#### WFL-06: Manual Completion Marking ✅
**User Story:** As an engineering user, I want to manually mark a request as completed after the requester receives the solution so that tracking is complete and E-ordering system integration point is documented.

**Acceptance Criteria:**
- [x] Engineering user can mark request as "Completed" only when status is "SendBackToRequester"
- [x] Manual completion is optional (engineering's choice for tracking)
- [x] Completion action is logged in audit trail
- [x] Completed requests are read-only (no further actions)
- [x] Requester can view completed status and all solution files

**Outcome:** Validated - Manual completion marking implemented

**Dependencies:** WFL-05

#### WFL-07: Requester and Engineer Cancellation ✅
**User Story:** As a requester or engineer, I want to cancel my own submission before it's approved so that mistakes can be corrected without resubmission.

**Acceptance Criteria:**
- [x] Requester can cancel their own request if status is "ImprovementRequest" (before any approvals)
- [x] Engineer can cancel their solution if status is "DesignCostEstimationApproval" (before any approvals)
- [x] Cancellation requires a comment explaining why
- [x] Cancelled items show status as "Cancelled" and are read-only
- [x] Cancellation is logged in audit trail with timestamp and reason
- [x] User can create new request/solution after cancellation

**Outcome:** Validated - Cancellation workflow for both requesters and engineers

**Dependencies:** REQ-01, WFL-05

### Visibility & Dashboard

#### DASH-01: Dashboard Views ✅
**User Story:** As a user, I want to see dashboard views of relevant requests so that I can quickly find what needs my attention.

**Acceptance Criteria:**
- [x] "My Requests" view shows all requests created by current user
- [x] "Pending My Approval" view shows requests awaiting approval from current user
- [x] "All Requests" view shows all requests in system (read-only for most users)
- [x] Each view shows request title, status, date, and current stage
- [x] Clicking request navigates to detail page (opens modal)
- [x] Views update in real-time (30-second auto-refresh)

**Outcome:** Validated - Three-tab dashboard with auto-refresh

**Dependencies:** REQ-01, WFL-03

#### DASH-02: Search and Filter ✅
**User Story:** As a user, I want to search and filter requests by status, department, and date so that I can find specific requests among many.

**Acceptance Criteria:**
- [x] User can filter by department (single-select dropdown)
- [x] User can filter by status (multi-select checkboxes)
- [x] User can filter by date range (start date, end date)
- [x] User can search by title (text search)
- [x] Filters can be combined (AND logic)
- [x] System shows number of matching requests
- [x] Filters persist across page navigation (per-tab state)

**Outcome:** Validated - Real-time filtering with TanStack Table ColumnFiltersState

**Dependencies:** DASH-01

#### DASH-03: Activity Timeline View ✅
**User Story:** As a user, I want to see a visual timeline of all events for a request so that I can understand complete history at a glance.

**Acceptance Criteria:**
- [x] Request detail page shows chronological timeline of all events
- [x] Each event shows: timestamp, user, action type, and details
- [x] Events include: creation, approvals, rejections, status changes, file uploads, cancellations
- [x] Timeline is scrollable for long histories
- [x] Comments are displayed inline with relevant events
- [x] Most recent events appear at top (with day grouping)

**Outcome:** Validated - Activity timeline with day grouping and chronological events

**Dependencies:** REQ-01, AUD-01

### Compliance & Audit

#### AUD-01: Immutable Audit Trail ✅
**User Story:** As a compliance officer, I want an immutable log of all actions so that every request has complete accountability and can be exported for external systems.

**Acceptance Criteria:**
- [x] System logs every action with: timestamp, user ID, action type, request ID, and details
- [x] Audit trail is append-only (no deletions or modifications)
- [x] Audit trail is enforced at database level (PostgreSQL triggers prevent tampering)
- [x] Admin can export audit trail for specific request (CSV or JSON)
- [x] Admin can export audit trail for date range (CSV or JSON)
- [x] Audit trail is queryable for dashboard and analytics
- [x] Exported audit logs are suitable for E-ordering system integration

**Outcome:** Validated - Immutable audit trail with PostgreSQL triggers and export functionality

**Dependencies:** REQ-01, WFL-04

### Configuration & Administration

#### CONF-01: User and Department Management UI ✅
**User Story:** As an admin, I want a UI to manage users and departments so that I can maintain system without database access.

**Acceptance Criteria:**
- [x] Admin screen lists all users with name, email, department, role, level
- [x] Admin can create new user (opens form with all fields)
- [x] Admin can edit existing user (opens form with pre-filled data)
- [x] Admin can deactivate user (soft delete, confirmation prompt)
- [x] Admin screen lists all departments with name and description
- [x] Admin can create new department
- [x] Admin can edit department details
- [x] Non-admins cannot access these screens (redirected)

**Outcome:** Validated - Complete admin management UI with edit functionality added in Phase 8

**Dependencies:** AUTH-02

#### CONF-02: Workflow Configuration Read-Only View ✅
**User Story:** As a non-admin user, I want to view approval workflow setup so that I understand how requests will be routed.

**Acceptance Criteria:**
- [x] All users can view approval hierarchy configuration
- [x] View is read-only (no editing for non-admins)
- [x] Display shows departments, levels, and assigned users
- [x] Display is clearly organized (tree or list format)
- [x] View is accessible from dashboard or help section

**Outcome:** Validated - Read-only workflow view at /dashboard/workflow

**Dependencies:** WFL-01

#### CONF-03: Drag-and-Drop Hierarchy Builder ✅
**User Story:** As an admin, I want to configure approval hierarchies using drag-and-drop UI so that I can easily set up and adjust workflows.

**Acceptance Criteria:**
- [x] Admin screen shows department selector and hierarchy builder
- [x] Hierarchy builder displays levels as draggable containers
- [x] Users are displayed as draggable items within levels
- [x] Admin can drag users between levels
- [x] Admin can reorder levels (drag to change sequence)
- [x] Admin can add/remove levels
- [x] Changes are saved explicitly (preview before save button - batch save workflow)
- [x] System validates hierarchy (must have at least one level)

**Outcome:** Validated - Drag-and-drop hierarchy builder with batch save and validation

**Dependencies:** WFL-01, CONF-02

#### CONF-04: Request Archival and Deletion ✅
**User Story:** As an admin, I want to archive or delete requests so that system stays organized and old data doesn't clutter views.

**Acceptance Criteria:**
- [x] Admin can archive any request (moves to archive, doesn't delete)
- [x] Admin can permanently delete any request (destructive, confirmation required)
- [x] Archived requests are excluded from default dashboard views
- [x] Admin can view archived requests via separate filter/view
- [x] Deletion action is logged in audit trail (including what was deleted)
- [x] Deleted requests are soft-deleted (marked as deleted, data preserved for audit)

**Outcome:** Validated - Archival with 90-day threshold and manual retention controls

**Dependencies:** AUD-01
