# Phase 4.1: Auto Refresh Overlay Window - Research

**Researched:** 2026-02-06
**Domain:** Next.js 15 App Router data refresh patterns with React 19 hooks
**Confidence:** HIGH

## Summary

This phase implements auto-refresh mechanisms for modal/dialog overlays in a Next.js 15 App Router application using React 19. The research focused on three core areas: (1) data refresh patterns after server actions, (2) loading state UX with skeleton components, and (3) stale data detection with conflict resolution.

The standard approach in Next.js 15 involves using `revalidatePath()` in server actions combined with React 19's new hooks (`useOptimistic`, `useActionState`, `useFormStatus`) for optimistic updates and loading states. For modal scenarios, the pattern is to refresh data by re-fetching in the same component rather than relying on cache invalidation alone, since modals are client components that may not automatically re-render when paths are revalidated.

Stale data detection uses optimistic locking with version numbers or timestamps - either stored in the database (Prisma `updatedAt` field) or via HTTP ETags. The UI shows a passive warning banner when conflicts are detected, allowing users to manually refresh rather than forcing automatic refresh that could disrupt their workflow.

**Primary recommendation:** Use explicit data re-fetch in modal's `loadRequest()` function after mutations, combined with `useActionState` for loading states and version-based conflict detection on user actions.

### Pattern 1: Modal Data Refresh After Action
**What:** Re-fetch modal data after user takes action (approve, reject, submit)
**When to use:** All user-triggered mutations in modal dialogs
**Example:**
```typescript
// Source: Next.js Official Docs - Getting Started: Updating Data
// https://nextjs.org/docs/app/getting-started/updating-data

'use client'

export function RequestDetailModal({ requestId, open, onOpenChange }) {
  const [request, setRequest] = useState(null)
  const [loading, setLoading] = useState(false)

  const loadRequest = async () => {
    setLoading(true)
    try {
      const data = await getRequest(requestId)
      setRequest(data)
    } finally {
      setLoading(false)
    }
  }

  useEffect(() => {
    if (open && requestId) {
      loadRequest()
    }
  }, [open, requestId])

  const handleApprove = async () => {
    await approveRequest(requestId)
    await loadRequest() // Re-fetch to get latest data
  }

  return (
    <Dialog open={open} onOpenChange={onOpenChange}>
      {loading ? <RequestDetailSkeleton /> : <RequestContent />}
    </Dialog>
  )
}
```

### Pattern 4: Optimistic Locking with Version Check
**What:** Detect stale data by comparing version/timestamp before mutation
**When to use:** All mutations that could have concurrent updates
**Example:**
```typescript
// Source: Optimistic Concurrency Control Guide
// https://medium.com/@gaddamnaveen192/understanding-optimistic-locking-a-key-to-handling-data-conflicts-63c086b850d5

// Server action
'use server'

export async function approveRequest(requestId: string, expectedUpdatedAt: Date) {
  const request = await prisma.request.findUnique({
    where: { id: requestId },
    select: { updatedAt: true, status: true }
  })

  // Check if data has changed since user loaded it
  if (request.updatedAt.getTime() !== new Date(expectedUpdatedAt).getTime()) {
    return {
      success: false,
      stale: true,
      message: 'This request was updated by another user.'
    }
  }

  // Proceed with update
  await prisma.request.update({
    where: { id: requestId },
    data: { status: 'Approved' }
  })

  return { success: true }
}

// Client component
const handleApprove = async () => {
  const result = await approveRequest(request.id, request.updatedAt)

  if (result.stale) {
    setShowStaleWarning(true)
  } else if (result.success) {
    await loadRequest()
  }
}
```

### Anti-Patterns to Avoid
- **Auto-refresh on timer without user action:** Creates race conditions with form inputs and wastes API calls
- **Relying solely on revalidatePath() in modals:** Modal is client component; cache invalidation doesn't trigger re-fetch automatically
- **Forcing refresh on conflict:** Interrupts user workflow; passive warning is better UX
- **Using router.refresh() in modals:** Refreshes entire page including background, not just modal content
- **Optimistic updates without rollback:** Must handle errors and revert to previous state

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Loading skeletons | Custom div shimmer animations | shadcn/ui Skeleton component | Handles animation, accessibility, consistent styling; matches existing UI components |
| Optimistic UI updates | Manual state tracking with flags | React 19 useOptimistic hook | Automatic rollback on error, handles transitions, prevents race conditions |
| Form pending state | Loading state with useEffect | React 19 useFormStatus hook | Built-in form context tracking, no prop drilling, works with Server Actions |
| Action result handling | Custom state + try/catch | React 19 useActionState hook | Progressive enhancement, integrates with form actions, manages pending state |
| Date comparison for conflicts | Manual timestamp math | date-fns isBefore/isAfter/isEqual | Edge cases (timezones, daylight saving), tested library |
| Stale data detection | Custom change tracking | Prisma updatedAt + version check | Built-in timestamp management, atomic updates, database-level consistency |

**Key insight:** React 19 introduced hooks specifically for forms and async actions. Using these built-in hooks provides better integration with Server Components/Actions and handles edge cases (concurrent updates, error boundaries, progressive enhancement) that custom solutions often miss.

## Common Pitfalls

### Pitfall 1: Modal Doesn't Re-render After revalidatePath()
**What goes wrong:** Calling `revalidatePath()` in server action but modal content doesn't update
**Why it happens:** Modal is a client component with local state; cache invalidation doesn't trigger component re-fetch
**How to avoid:** Always call explicit data fetch function after server action completes in client component
**Warning signs:** User clicks action button, server action succeeds, but UI shows stale data until modal is closed/reopened

### Pitfall 2: Race Condition Between Auto-Refresh and Form Input
**What goes wrong:** User typing in textarea when auto-refresh triggers, losing their input
**Why it happens:** Refresh replaces component state without checking if user is interacting with form
**How to avoid:** Track form focus/blur state; block refresh when any form field has focus
**Warning signs:** User reports "my text disappeared while I was typing"

### Pitfall 3: Showing Loading State for Cached Data
**What goes wrong:** Skeleton appears even when data loads instantly from cache
**Why it happens:** Setting loading=true before checking if data is in React state/cache
**How to avoid:** Only show skeleton after 200ms delay or use Suspense boundaries for streaming
**Warning signs:** UI flashes skeleton on every modal open, feels janky

### Pitfall 4: Version Mismatch False Positives from Date Precision
**What goes wrong:** Conflict warning appears even when no real conflict exists
**Why it happens:** JavaScript Date loses milliseconds in serialization or database rounds timestamps
**Why it happens:** Prisma `@updatedAt` uses database server time; comparing with client-sent Date can have millisecond differences
**How to avoid:** Use numeric version field OR compare timestamps with 1-second tolerance OR use `.toISOString()` for string comparison
**Warning signs:** Stale data warnings appear on first action after page load

### Pitfall 5: Infinite Refresh Loop
**What goes wrong:** Component continuously re-fetches data in endless loop
**Why it happens:** useEffect dependency array includes objects/functions that are recreated on every render
**How to avoid:** Only include primitive dependencies (requestId, open) in useEffect; use useCallback for functions
**Warning signs:** Network tab shows repeated identical requests; browser becomes unresponsive

### Pitfall 6: Server Action Called After Component Unmounts
**What goes wrong:** setState called on unmounted component after async server action completes
**Why it happens:** User closes modal before server action finishes; component unmounted but async continues
**How to avoid:** Use cleanup function in useEffect OR check mounted state before setState OR use React 19's automatic cleanup
**Warning signs:** Console warnings "Can't perform React state update on unmounted component"

### Complete Modal Refresh Pattern
```typescript
// Source: Next.js Official Docs + React 19 Hooks
// https://nextjs.org/docs/app/getting-started/updating-data
// https://react.dev/reference/react/useOptimistic

'use client'

import { useState, useEffect, useCallback } from 'react'
import { Dialog, DialogContent } from '@/components/ui/dialog'
import { getRequest } from '@/server-actions/requests'
import { approveRequest } from '@/server-actions/approvals'
import { RequestDetailSkeleton } from '@/components/loading/request-detail-skeleton'
import { StaleDataBanner } from '@/components/requests/stale-data-banner'

interface RequestDetailModalProps {
  requestId: string
  open: boolean
  onOpenChange: (open: boolean) => void
}

export function RequestDetailModal({
  requestId,
  open,
  onOpenChange,
}: RequestDetailModalProps) {
  const [request, setRequest] = useState<any>(null)
  const [loading, setLoading] = useState(true)
  const [isFormActive, setIsFormActive] = useState(false)
  const [showStaleWarning, setShowStaleWarning] = useState(false)

  // Memoize loadRequest to prevent infinite loops in useEffect
  const loadRequest = useCallback(async () => {
    if (!requestId) return

    setLoading(true)
    setShowStaleWarning(false)

    try {
      const data = await getRequest(requestId)
      setRequest(data)
    } catch (error) {
      console.error('Failed to load request:', error)
    } finally {
      setLoading(false)
    }
  }, [requestId])

  // Load request when modal opens or requestId changes
  useEffect(() => {
    if (open && requestId) {
      loadRequest()
    }
  }, [open, requestId, loadRequest])

  // Handle approval with conflict detection
  const handleApprove = async () => {
    if (!request) return

    // Pass current updatedAt for optimistic locking
    const result = await approveRequest(request.id, request.updatedAt)

    if (result.stale) {
      // Show warning banner instead of auto-refresh
      setShowStaleWarning(true)
    } else if (result.success) {
      // Refresh modal to show latest data
      await loadRequest()
    }
  }

  // Block refresh while user is typing
  const handleFormFocus = () => setIsFormActive(true)
  const handleFormBlur = () => setIsFormActive(false)

  return (
    <Dialog open={open} onOpenChange={onOpenChange}>
      <DialogContent className="max-w-3xl max-h-[90vh]">
        {loading ? (
          <RequestDetailSkeleton />
        ) : !request ? (
          <div className="text-center py-8 text-gray-500">
            Request not found
          </div>
        ) : (
          <div className="space-y-4">
            {/* Stale data warning */}
            {showStaleWarning && (
              <StaleDataBanner onRefresh={loadRequest} />
            )}

            {/* Request content */}
            <div>
              <h2>{request.title}</h2>
              <p>{request.description}</p>
            </div>

            {/* Action form with focus tracking */}
            <form
              onFocus={handleFormFocus}
              onBlur={handleFormBlur}
            >
              <textarea name="comments" />
              <button onClick={handleApprove}>Approve</button>
            </form>
          </div>
        )}
      </DialogContent>
    </Dialog>
  )
}
```

### Server Action with Optimistic Locking
```typescript
// Source: Optimistic Locking Best Practices
// https://www.baeldung.com/jpa-optimistic-locking

'use server'

import { revalidatePath } from 'next/cache'
import { prisma } from '@/lib/prisma'

export async function approveRequest(
  requestId: string,
  expectedUpdatedAt: string | Date
) {
  // Fetch current state
  const request = await prisma.request.findUnique({
    where: { id: requestId },
    select: {
      id: true,
      status: true,
      updatedAt: true
    }
  })

  if (!request) {
    return { success: false, error: 'Request not found' }
  }

  // Compare timestamps (with 1-second tolerance for DB precision)
  const currentTime = request.updatedAt.getTime()
  const expectedTime = new Date(expectedUpdatedAt).getTime()
  const timeDiff = Math.abs(currentTime - expectedTime)

  if (timeDiff > 1000) {
    // Data has changed since user loaded it
    return {
      success: false,
      stale: true,
      message: 'This request was updated by another user.',
      currentUpdatedAt: request.updatedAt
    }
  }

  // Perform update
  await prisma.request.update({
    where: { id: requestId },
    data: {
      status: 'Approved',
      // updatedAt is auto-updated by Prisma @updatedAt
    }
  })

  // Revalidate path for other pages (not modal)
  revalidatePath('/dashboard')

  return { success: true }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| useFormState (React 18) | useActionState (React 19) | Dec 2024 | Renamed for clarity; API is identical |
| router.refresh() for modals | Explicit data re-fetch | Next.js 13+ | router.refresh() refreshes entire page; explicit fetch only refreshes modal |
| Custom optimistic state | useOptimistic hook | React 19 (Dec 2024) | Built-in rollback, transition handling, cleaner code |
| Pessimistic locking | Optimistic locking with version check | Industry standard 2020+ | Better scalability, no database locks, RESTful pattern |
| Timestamp-based versioning | Numeric version field | Industry standard | More reliable, avoids millisecond precision issues |

**Deprecated/outdated:**
- **useFormState**: Renamed to `useActionState` in React 19 (Dec 2024) - functionality identical, just naming change
- **router.refresh() in modals**: Still works but refreshes entire page including background; use explicit data fetch for modal-only refresh
- **revalidateTag without explicit fetch in client components**: Cache invalidation doesn't trigger client component re-renders; must call fetch function

## Open Questions

Things that couldn't be fully resolved:

1. **Optimal polling interval for notification-based refresh**
   - What we know: Project uses 30-second polling for notifications (established in 04-07)
   - What's unclear: Should auto-refresh use same polling interval or rely only on user actions?
   - Recommendation: Start with user-action-only refresh (per CONTEXT.md); can add polling later if UAT feedback requests it

2. **Cross-tab synchronization**
   - What we know: CONTEXT.md specifies refresh only in tab where action occurred
   - What's unclear: Should we use BroadcastChannel API for future cross-tab sync?
   - Recommendation: Implement single-tab refresh first; add cross-tab sync in Phase 5 if needed for dashboard widgets

3. **Skeleton delay threshold**
   - What we know: Industry standard is 200ms perceived delay threshold
   - What's unclear: Should we use setTimeout delay or Suspense boundary for skeleton display?
   - Recommendation: Use simple loading state without delay for modal; fast server actions (<200ms) won't show noticeable flash

4. **Version field vs timestamp for conflict detection**
   - What we know: Prisma auto-manages `updatedAt` field; comparing timestamps has precision issues
   - What's unclear: Should we add dedicated numeric `version` field to Request model?
   - Recommendation: Use `updatedAt` with 1-second tolerance initially; add `version` field if false positives occur
