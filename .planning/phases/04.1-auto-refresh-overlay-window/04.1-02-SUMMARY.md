## Accomplishments

- Added `isStaleData` helper function to both approval server action files with 1-second tolerance
- Updated `approveRequest` and `rejectRequest` to accept optional `expectedUpdatedAt` parameter
- Updated `approveSolution`, `rejectSolution`, `approveFinalApproval`, and `rejectFinalApproval` to accept optional `expectedUpdatedAt` parameter
- All functions return `{ success: false, stale: true, message: '...' }` when data has been modified by another user
- Backward compatible - existing calls without `expectedUpdatedAt` work unchanged

## Decisions Made

- Used `request.updatedAt` as source of truth for all stale data checks (including solution/final approvals) since the conflict detection is about the overall request state that users are viewing
- 1-second tolerance prevents false positives from database timestamp precision issues (PostgreSQL timestamps have microsecond precision but client-side Date objects may differ)
- Optional `expectedUpdatedAt` parameter ensures backward compatibility - existing UI calls continue working without modification
