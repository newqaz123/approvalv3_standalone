# Prior Phase Decisions

From Phase 1 (Foundation & Authentication):
- **User creation**: Admin creates users via UI, which calls createUser() Server Action
- **Wrong invitation method**: Current code uses `verifyUserEmail()` (Gap #4 from v1-MILESTONE-AUDIT.md)
- **Transaction rollback**: Added in plan 01.2-01 (separate concern)

**Current problem (from v1-MILESTONE-AUDIT.md Gap #4):**
```typescript
// src/server-actions/users.ts (line 97)
await clerkClient.users.verifyUserEmail(clerkUser.id)
```

**Why this is wrong:**
- `verifyUserEmail()` marks an email address as verified (for compliance)
- Does NOT send invitation emails
- Used for email verification flows, not user invitations
- New users never receive invitation to complete signup

**Correct approach (from RESEARCH.md Pattern 3):**
```typescript
// Option 1: Don't set password - Clerk sends invitation automatically
const clerkUser = await clerkClient.users.createUser({
  emailAddress: [input.email],
  firstName: input.name.split(' ')[0],
  lastName: input.name.split(' ').slice(1).join(' '),
  publicMetadata: {
    role: input.role,
    departmentId: input.departmentId,
  },
  // No password field = invitation email sent automatically
})
```

**Why this works:**
- When `password` field is omitted from `createUser()`, Clerk automatically sends invitation email
- User receives email with link to complete signup and set password
- Simpler than calling separate invitation API
- Standard pattern for admin-created users (Pitfall #4 from RESEARCH.md)
</context>
