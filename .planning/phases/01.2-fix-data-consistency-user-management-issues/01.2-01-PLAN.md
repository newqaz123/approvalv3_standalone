- **Database-backed role checks**: Server Actions use Prisma for authorization (Phase 01.1)

**Required fix from RESEARCH.md (Pattern 3):**
```typescript
let clerkUser: any

try {
  clerkUser = await clerkClient.users.createUser({...})
} catch (error) {
  throw new Error(`Failed to create Clerk user: ${error}`)
}

try {
  const user = await prisma.user.create({...})
  revalidatePath('/admin/users')
  return user
} catch (error) {
  // ✅ Rollback Clerk user if Prisma fails
  await clerkClient.users.deleteUser(clerkUser.id)
  throw new Error(`Failed to create user record: ${error}`)
}
```

**Code structure:**
```typescript
// Around line 85-112
let clerkUser: any

try {
  // Step 1: Create Clerk user
  clerkUser = await clerkClient.users.createUser({
    emailAddress: [input.email],
    firstName: input.name.split(' ')[0],
    lastName: input.name.split(' ').slice(1).join(' '),
    publicMetadata: {
      role: input.role,
      departmentId: input.departmentId,
    },
  })

  // Note: verifyUserEmail() will be fixed in plan 01.2-02

} catch (error) {
  console.error('❌ Failed to create Clerk user:', error)
  throw new Error(`Failed to create Clerk user: ${error}`)
}

try {
  // Step 2: Create Prisma record
  const user = await prisma.user.create({
    data: {
      id: clerkUser.id,
      email: input.email,
      name: input.name,
      departmentId: input.departmentId,
      role: input.role,
      isActive: true,
    },
  })

  revalidatePath('/admin/users')
  return user

} catch (error) {
  // Step 3: Rollback Clerk user if Prisma fails
  console.error('❌ Prisma create failed, rolling back Clerk user:', clerkUser.id, error)
  await clerkClient.users.deleteUser(clerkUser.id)
  throw new Error(`Failed to create user record: ${error}`)
}
```

**WHY this pattern:**
- Prevents orphaned Clerk users (Pitfall #3 from RESEARCH.md)
- Dual-write systems require explicit rollback when second write fails
- Clerk user created first because Prisma depends on clerkUser.id
- Logging helps debug transaction failures

**DO NOT:**
- Use Prisma `$transaction()` API - not applicable here (Clerk is external API)
- Remove `verifyUserEmail()` call - fixed in separate plan (01.2-02)
- Modify other functions (updateUser, deactivateUser) - only createUser needs rollback
