# Phase 01.2: Fix Data Consistency & User Management Issues - Research

**Researched:** 2026-01-31
**Domain:** Clerk Authentication Integration, Prisma ORM, Dual-Write Systems
**Confidence:** HIGH

## Summary

This phase addresses critical gaps in the Clerk + Prisma dual-write architecture that prevent production readiness. The core issue is inconsistent state management between Clerk (auth provider) and Prisma (application database), affecting user signup flows, role-based access control, and user invitation patterns.

**Key findings:**
1. **Clerk webhooks are the standard pattern** for syncing user data to external databases - webhook handlers should create database records idempotently
2. **JWT metadata caching is fundamental** to Clerk's design - session tokens cache `publicMetadata` at token issuance time (default 60s TTL), making database-backed role checks necessary for immediate consistency
3. **Dual-write systems require transaction rollback patterns** - create external resources first (Clerk), then database (Prisma), with cleanup if database fails
4. **User invitations should use Clerk's invitation API** (`createInvitation()`), not `verifyUserEmail()` which is for email verification flows
5. **Prisma provides robust transaction APIs** for handling multi-step operations with automatic rollback

**Primary recommendation:** Use webhook-based user creation for self-signups, database-backed role checks for access control, and transaction rollback pattern for admin-created users.

### Pattern 1: Webhook-Based User Creation (Self-Signup Flow)

**What:** Clerk webhooks push `user.created` events to your API when users sign up. Your webhook handler creates a Prisma record to mirror the Clerk user.

**When to use:** All self-registration flows (email/password, OAuth, magic links).

**Why:** Ensures every Clerk user has a corresponding Prisma record. Real-time sync. Idempotent by design (Clerk retries on failure).

**Example:**
```typescript
// Source: https://github.com/clerk/clerk-docs/blob/main/docs/guides/development/webhooks/syncing.mdx
import { verifyWebhook } from '@clerk/nextjs/webhooks'
import { NextRequest, NextResponse } from 'next/server'
import prisma from '@/lib/prisma'

export async function POST(req: NextRequest) {
  try {
    // Step 1: Verify webhook signature (prevents spoofing)
    const evt = await verifyWebhook(req)
    const eventType = evt.type

    // Step 2: Handle user creation
    if (eventType === 'user.created') {
      const { id, email_addresses, first_name, last_name } = evt.data
      const email = email_addresses[0]?.email_address

      if (!email) {
        return new NextResponse('Invalid payload', { status: 400 })
      }

      // Step 3: Create Prisma user record
      // Note: Clerk metadata already updated in this event
      await prisma.user.create({
        data: {
          id: id,
          email: email,
          name: `${first_name} ${last_name}`.trim() || 'User',
          role: 'general_dept', // Default role for self-signups
          isActive: true,
        },
      })

      console.log(`✅ User created via webhook: ${id} (${email})`)
    }

    return NextResponse.json({ success: true }, { status: 200 })

  } catch (error) {
    console.error('❌ Webhook error:', error)

    // Idempotency: Return 200 if user already exists
    if (error instanceof Error && error.message.includes('Unique constraint')) {
      return NextResponse.json({ success: true, idempotent: true }, { status: 200 })
    }

    // Return 400 to prevent Clerk retries
    return NextResponse.json({ error: 'Webhook processing failed' }, { status: 400 })
  }
}
```

**Key points:**
- `verifyWebhook()` handles signature verification automatically using `CLERK_WEBHOOK_SECRET`
- Unique constraint violations should return 200 (idempotent)
- Webhook failures should return 4xx to prevent retries (handled via Svix)

### Pattern 2: Database-Backed Role Checks (Access Control)

**What:** Query Prisma for user role on every protected operation instead of relying on JWT metadata.

**When to use:** All Server Actions that check permissions. Middleware can use JWT for performance (documented as "may be stale").

**Why:** Clerk JWT tokens cache `publicMetadata` from token issuance time. When an admin changes a user's role, the user's JWT still contains the old role until token expires (up to 60 seconds). Database queries always return current role.

**Example:**
```typescript
// src/lib/auth.ts
import { auth } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'

export async function requireAdmin() {
  const { userId } = await auth()

  if (!userId) {
    return null
  }

  // ✅ Check Prisma for current role (always reflects latest changes)
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { role: true, isActive: true },
  })

  // Verify user exists, is active, and has admin role
  if (!user || !user.isActive || user.role !== 'admin') {
    return null
  }

  return userId
}
```

**Middleware pattern (with documented staleness):**
```typescript
// src/middleware.ts
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server'
import { NextResponse } from 'next/server'

const isAdminRoute = createRouteMatcher(['/admin(.*)'])

export default clerkMiddleware(async (auth, req) => {
  const { sessionClaims, userId } = await auth()

  if (isAdminRoute(req)) {
    if (!userId) {
      const signInUrl = new URL('/sign-in', req.url)
      return NextResponse.redirect(signInUrl)
    }

    // Check role from JWT metadata (may be up to 60s stale, but fast)
    // Actual enforcement happens in Server Actions via requireAdmin()
    if (sessionClaims?.metadata?.role !== 'admin') {
      return NextResponse.redirect(new URL('/dashboard', req.url))
    }
  }
})
```

**Key points:**
- JWT metadata checks in middleware are "best effort" for performance
- Server Actions MUST use database-backed checks for security
- This hybrid approach balances edge performance (middleware) with data consistency (Server Actions)

### Pattern 3: Dual-Write with Transaction Rollback (Admin User Creation)

**What:** Create Clerk user first, then Prisma record. If Prisma fails, delete the Clerk user to prevent inconsistent state.

**When to use:** Admin-created users where both systems must stay in sync.

**Why:** Clerk is the source of truth for authentication. If Prisma fails, we must rollback Clerk to avoid users receiving invitations but being unable to access the app.

**Example:**
```typescript
// src/server-actions/users.ts
import { clerkClient } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'

export async function createUser(input: CreateUserInput) {
  await requireAdmin()

  let clerkUser: any

  try {
    // Step 1: Create Clerk user (no password = invitation email sent automatically)
    clerkUser = await clerkClient.users.createUser({
      emailAddress: [input.email],
      firstName: input.name.split(' ')[0],
      lastName: input.name.split(' ').slice(1).join(' '),
      publicMetadata: {
        role: input.role,
        departmentId: input.departmentId,
      },
    })

  } catch (error) {
    throw new Error(`Failed to create Clerk user: ${error}`)
  }

  try {
    // Step 2: Create Prisma record
    const user = await prisma.user.create({
      data: {
        id: clerkUser.id,
        email: input.email,
        name: input.name,
        departmentId: input.departmentId,
        role: input.role,
        isActive: true,
      },
    })

    revalidatePath('/admin/users')
    return user

  } catch (error) {
    // Step 3: Rollback Clerk user if Prisma fails
    await clerkClient.users.deleteUser(clerkUser.id)
    throw new Error(`Failed to create user record: ${error}`)
  }
}
```

**Key points:**
- Clerk user created without password triggers automatic invitation email
- Try/catch around Prisma create ensures rollback on failure
- Rollback prevents inconsistent state (Clerk user exists but Prisma record missing)

### Anti-Patterns to Avoid

- **Using `verifyUserEmail()` for invitations:** This method marks an email as verified, not send invitations. Use `createUser()` without password instead.
- **Relying solely on JWT metadata for permissions:** JWTs cache metadata for up to 60 seconds. Always verify critical permissions in Server Actions using database queries.
- **Returning 5xx errors from webhooks:** Clerk will retry indefinitely. Return 4xx for permanent failures, 2xx for idempotent duplicates.
- **Commenting out security checks:** Never bypass role checks with "TODO" comments. Use feature flags or database-backed checks immediately.

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Webhook signature verification | Custom crypto logic | `verifyWebhook()` from `@clerk/nextjs/webhooks` | Clerk handles signature validation, timestamp checks, replay prevention |
| Idempotency for webhooks | Custom deduplication tables | Prisma unique constraint + 200 on duplicate | Simpler, leverages database constraints |
| Transaction rollback | Custom cleanup scripts | Try/catch with `deleteUser()` | Explicit, atomic, easier to reason about |
| Role checks from JWT | Custom token parsing | Database query via Prisma | Always current, no caching issues |

**Key insight:** Clerk's webhooks are designed for idempotent database synchronization. Don't reinvent the event bus pattern.

## Common Pitfalls

### Pitfall 1: Missing Prisma User Records

**What goes wrong:** Self-signed up users have Clerk accounts but no Prisma records, breaking any feature that queries the User table.

**Why it happens:** Webhook handler only updates Clerk metadata but doesn't create Prisma records. Webhook completes successfully, so Clerk doesn't retry.

**How to avoid:**
1. Add `prisma.user.create()` to `user.created` webhook handler
2. Test with Clerk Dashboard webhook tester
3. Verify Prisma records exist after signup

**Warning signs:** User can sign in but doesn't appear in `/admin/users`, permission checks fail inconsistently.

### Pitfall 2: Stale JWT Metadata Causing Permission Bypass

**What goes wrong:** User role changed in Prisma but JWT still contains old role, allowing access to protected routes OR denying access to newly granted permissions.

**Why it happens:** Clerk JWT tokens cache `publicMetadata` at token issuance time. Tokens refresh every 60 seconds by default.

**How to avoid:**
1. Use database-backed role checks in Server Actions (requireAdmin)
2. Document JWT checks in middleware as "best effort" for performance
3. Never rely solely on JWT metadata for security-critical operations

**Warning signs:** Role changes don't take effect immediately, users report "I just changed their role but they still can/can't access".

### Pitfall 3: Orphaned Clerk Users After Prisma Failure

**What goes wrong:** Clerk user created and invitation sent, but Prisma create fails. User receives email but can't access app. Manual cleanup required in Clerk Dashboard.

**Why it happens:** Dual-write system without transaction rollback. If Prisma fails after Clerk succeeds, state is inconsistent.

**How to avoid:**
1. Wrap Prisma create in try/catch
2. Call `clerkClient.users.deleteUser()` in catch block
3. Test with invalid Prisma data (duplicate email, missing department)

**Warning signs:** Users report "I got the invitation email but it says invalid user when I click the link".

### Pitfall 4: Wrong Email Invitation Method

**What goes wrong:** Using `verifyUserEmail()` which marks email as verified, doesn't send invitation email.

**Why it happens:** Confusion between email verification and user invitation flows.

**How to avoid:**
1. Use `createUser()` without `password` field - Clerk sends invitation automatically
2. OR use `clerkClient.invitations.createInvitation()` for invitation-only flow
3. Never use `verifyUserEmail()` for new user invitations

**Warning signs:** Test invitations don't arrive, or users report "I never got the email but my account exists".

### Pitfall 5: Webhook Infinite Retry Loop

**What goes wrong:** Webhook handler returns 5xx error on every retry. Clerk retries indefinitely (via Svix), creating logs and spam.

**Why it happens:** Permanent error (e.g., missing database column) but returning 500 instead of 4xx.

**How to avoid:**
1. Return 400 for permanent failures (invalid payload, configuration errors)
2. Return 200 for idempotent duplicates (unique constraint violations)
3. Only return 500 for transient failures (network timeouts)

**Warning signs:** Clerk Dashboard shows "Retrying" status, webhook logs show thousands of attempts.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual webhook verification | `verifyWebhook()` helper | Clerk 4.0+ | Simplified code, automatic signature validation |
| JWT-only role checks | Database-backed checks in Server Actions | 2024+ | Fixes stale metadata issue, immediate consistency |
| Manual transaction management | Try/catch with explicit rollback | Always applicable | Prevents dual-write inconsistencies |
| Email verification for invitations | `createUser()` without password | Clerk 5.0+ | Automatic invitation emails, simpler API |

**Deprecated/outdated:**
- Manual webhook signature verification: Use `verifyWebhook()` from `@clerk/nextjs/webhooks`
- JWT-only permission checks: Combine with database checks for critical operations
- `verifyUserEmail()` for invitations: Use `createUser()` without password or `invitations.createInvitation()`

## Open Questions

1. **Clerk Invitations API vs createUser without password**
   - What we know: Both approaches send invitation emails. `createUser()` creates user immediately, `invitations.createInvitation()` creates pending invitation.
   - What's unclear: Which approach better matches the application's user onboarding flow.
   - Recommendation: Use `createUser()` without password for simplicity. User record exists immediately upon accepting invitation. If invitation expiration/cancellation is needed, use `invitations.createInvitation()`.

2. **Webhook testing in development**
   - What we know: Clerk Dashboard has webhook testing tab that sends example events.
   - What's unclear: How to test webhooks with localhost during development (tunneling vs. ngrok vs. Clerk Dev Sessions).
   - Recommendation: Use Clerk Dashboard's webhook testing feature with deployed endpoint, or use local tunneling (ngrok/Cloudflare Tunnel) for local development.
