<task type="auto">
  <name>Task 1: Implement Hierarchy Persistence Logic</name>
  <files>src/server-actions/hierarchy.ts</files>
  <action>
    Create `src/server-actions/hierarchy.ts`:
    1. `getDepartmentHierarchy(departmentId)`:
       - Fetch all users in department.
       - Fetch all `DepartmentApprover`s for this department (external users).
       - Merge into a unified list of `{ user, level, isExternal }`.
    2. `updateHierarchy(departmentId, updates)`:
       - Validate inputs before saving:
         - **Continuous Levels:** Ensure levels are sequential starting from 1 (e.g., 1, 2, 3) with no gaps (e.g., cannot have 1, 3).
         - **Empty Levels:** Ensure every defined level has at least one approver.
         - **Cycle Detection:** Check for approval cycles (e.g., User A approves User B, B approves A). If a cycle is detected, throw a specific error.
       - `updates` is array of `{ userId, level }`.
       - For each update:
         - If User belongs to Department -> Update `User.level`.
         - If User is External -> Upsert `DepartmentApprover`.
         - If Level is null (removed) -> Set `User.level = null` OR delete `DepartmentApprover`.
       - Return specific error messages if validation fails (e.g. "Level 2 is empty", "Cycle detected involving User X").
  </action>
  <verify>
    Call server actions from a test script or console.
    Verify database updates correctly reflect "Global" vs "External" logic.
  </verify>
  <done>
    Server actions for fetching and saving hierarchy data exist.
  </done>
</task>
