# Phase 01.1: Fix Critical Authentication & Security Gaps - Research

**Researched:** 2025-01-31
**Domain:** Clerk Webhooks, Prisma Integration, Database-Backed Authorization
**Confidence:** HIGH

## Summary

This phase addresses two critical security gaps identified in the v1 Milestone Audit: (1) broken user signup flow where webhook doesn't create Prisma records, and (2) bypassed role-based access control allowing any authenticated user to access admin routes. Research focused on Clerk webhook best practices, database-backed authorization patterns, and error handling strategies.

The standard approach for Clerk + Prisma integrations is to use webhooks for user creation and database-backed authorization checks for role validation. Clerk's `verifyWebhook()` utility provides signature verification using Svix headers. For role checks, the recommended pattern is to query Prisma directly rather than relying on JWT metadata, which can be stale until token refresh (default 60 seconds).

**Primary recommendation:** Use `verifyWebhook()` from `@clerk/nextjs/webhooks` for secure webhook handling, implement database-backed role checks via Prisma for real-time authorization, and wrap dual-write operations in try/catch blocks with rollback logic.

### Pattern 1: Clerk Webhook with Prisma User Creation

**What:** Webhook handler that creates Prisma records on `user.created` events
**When to use:** Self-signup flow where users register via Clerk UI
**Why:** Ensures every Clerk user has corresponding Prisma record for application data

**Example:**
```typescript
// Source: /clerk/clerk-docs - Webhooks syncing guide
import { verifyWebhook } from '@clerk/nextjs/webhooks'
import { NextRequest } from 'next/server'
import prisma from '@/lib/prisma'

export async function POST(req: NextRequest) {
  try {
    // Verify webhook signature (Svix headers)
    const evt = await verifyWebhook(req)

    const { id } = evt.data
    const eventType = evt.type

    // Handle user creation
    if (evt.type === 'user.created') {
      const { id, email_addresses, first_name, last_name } = evt.data
      const email = email_addresses[0]?.email_address

      // Create Prisma record
      await prisma.user.create({
        data: {
          id: id,
          email: email,
          name: `${first_name} ${last_name}`.trim(),
          role: 'general_dept',
          isActive: true,
        },
      })

      console.log(`User created: ${id}`)
    }

    return new Response('Webhook received', { status: 200 })
  } catch (err) {
    console.error('Error verifying webhook:', err)
    return new Response('Error verifying webhook', { status: 400 })
  }
}
```

**Key Points:**
- `verifyWebhook()` handles signature verification automatically using `CLERK_WEBHOOK_SECRET`
- Must return 200 status on success to prevent retries (Clerk uses Svix retry schedule)
- Webhook data includes all user fields (email, name, etc.)
- Create user with default role (`general_dept` for self-signups)

### Pattern 2: Database-Backed Authorization

**What:** Query Prisma for current role instead of reading from JWT metadata
**When to use:** Role-based access control where admin can change user roles
**Why:** Prisma always reflects latest role changes; JWT metadata cached for 60s

**Example:**
```typescript
// Source: /clerk/clerk-docs - Force token refresh guide
import { auth } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'

export async function requireAdmin() {
  const { userId } = await auth()
  if (!userId) {
    return null
  }

  // Check Prisma for current role (always reflects latest changes)
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { role: true },
  })

  if (!user || user.role !== 'admin') {
    return null
  }

  return userId
}
```

**Key Points:**
- Bypasses stale JWT metadata issue (tokens cache `publicMetadata` from issuance)
- Provides real-time role checks after admin changes user role
- Small performance trade-off (one DB query per auth check)
- Can be cached with React cache() if performance becomes issue

### Pattern 3: Dual-Write with Rollback

**What:** Create Clerk user and Prisma record with rollback on failure
**When to use:** Admin-created users where both systems must stay in sync
**Why:** Prevents orphaned Clerk users or missing Prisma records

**Example:**
```typescript
// Source: /prisma/prisma - Error handling patterns
import { clerkClient } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'

export async function createUser(input: CreateUserInput) {
  let clerkUser: any

  try {
    // Step 1: Create Clerk user
    clerkUser = await clerkClient.users.createUser({
      emailAddress: [input.email],
      firstName: input.name.split(' ')[0],
      lastName: input.name.split(' ').slice(1).join(' '),
      publicMetadata: {
        role: input.role,
        departmentId: input.departmentId,
      },
    })
  } catch (error) {
    throw new Error(`Failed to create Clerk user: ${error}`)
  }

  try {
    // Step 2: Create Prisma record
    const user = await prisma.user.create({
      data: {
        id: clerkUser.id,
        email: input.email,
        name: input.name,
        departmentId: input.departmentId,
        role: input.role,
        isActive: true,
      },
    })

    return user

  } catch (error) {
    // Rollback: Delete Clerk user if Prisma fails
    await clerkClient.users.deleteUser(clerkUser.id)
    throw new Error(`Failed to create user record: ${error}`)
  }
}
```

**Key Points:**
- Clerk user created first (source of truth)
- Prisma record created second
- If Prisma fails, Clerk user deleted to maintain consistency
- Prevents users receiving invitation emails but unable to access app

### Pattern 4: Middleware with Database-Backed Checks

**What:** Protect routes by checking Prisma for user roles
**When to use:** Route-level authorization for admin-only pages
**Why:** Enforces real-time role checks at edge

**Example:**
```typescript
// Source: /clerk/clerk-docs - Middleware protection
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server'
import { NextResponse } from 'next/server'
import prisma from '@/lib/prisma'

const isAdminRoute = createRouteMatcher(['/admin(.*)'])

export default clerkMiddleware(async (auth, req) => {
  const { userId } = await auth()

  // Protect admin routes
  if (isAdminRoute(req)) {
    if (!userId) {
      const signInUrl = new URL('/sign-in', req.url)
      return NextResponse.redirect(signInUrl)
    }

    // Database-backed role check
    const user = await prisma.user.findUnique({
      where: { id: userId },
      select: { role: true },
    })

    if (!user || user.role !== 'admin') {
      return NextResponse.redirect(new URL('/dashboard', req.url))
    }
  }
})
```

**Key Points:**
- Middleware runs at edge before page renders
- Can access Prisma via singleton import
- Redirects non-admins to `/dashboard`
- Remove commented-out bypass code

### Anti-Patterns to Avoid

- **JWT metadata for role checks:** Metadata cached for 60s, becomes stale after role changes
- **Commented-out security code:** Bypasses with TODO comments should use feature flags instead
- **Webhooks without error handling:** Unhandled exceptions cause 500 errors, triggering unnecessary retries
- **Dual-write without rollback:** Orphaned Clerk users break user experience
- **Manual signature verification:** Use `verifyWebhook()` instead of parsing Svix headers manually

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Webhook signature verification | Manual Svix header parsing | `verifyWebhook()` from `@clerk/nextjs/webhooks` | Handles timestamp, signature, ID validation automatically |
| JWT metadata refresh logic | Custom token invalidation | Database-backed role checks | Tokens cache metadata; Prisma always current |
| Webhook retry logic | Custom exponential backoff | Clerk's built-in Svix retries | Svix handles retry schedule automatically |
| Transaction safety | Manual cleanup code | Try/catch with rollback | Simpler, easier to reason about |

**Key insight:** Clerk provides built-in webhook verification and retry logic via Svix. Don't rebuild these. For authorization, database checks are more reliable than JWT metadata caching.

## Common Pitfalls

### Pitfall 1: Webhook Fails Silently

**What goes wrong:** Webhook handler throws error, returns 500, Clerk retries infinitely
**Why it happens:** Missing try/catch around webhook logic, unhandled exceptions
**How to avoid:** Wrap webhook handler in try/catch, return appropriate status codes
**Warning signs:** Clerk Dashboard shows "Failed" webhook attempts with status 500

**Example:**
```typescript
// BAD - Unhandled exception
export async function POST(req: NextRequest) {
  const evt = await verifyWebhook(req)
  await prisma.user.create({ ... })  // Throws if duplicate user
  return new Response('Success', { status: 200 })
}

// GOOD - Error handling
export async function POST(req: NextRequest) {
  try {
    const evt = await verifyWebhook(req)

    if (evt.type === 'user.created') {
      await prisma.user.create({ ... })
    }

    return new Response('Success', { status: 200 })
  } catch (err) {
    console.error('Webhook error:', err)
    return new Response('Error', { status: 400 })  // Prevents retries
  }
}
```

### Pitfall 2: Stale JWT Metadata After Role Changes

**What goes wrong:** Admin changes user role, user still has old role for up to 60 seconds
**Why it happens:** Clerk JWT tokens cache `publicMetadata` from when token was issued
**How to avoid:** Use database-backed role checks instead of JWT metadata
**Warning signs:** Users report "role changes not taking effect immediately"

**Example:**
```typescript
// BAD - Reads stale metadata
export async function requireAdmin() {
  const { sessionClaims } = await auth()
  if (sessionClaims?.metadata?.role !== 'admin') {
    return null
  }
  return userId
}

// GOOD - Checks current database state
export async function requireAdmin() {
  const { userId } = await auth()
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { role: true },
  })
  if (!user || user.role !== 'admin') {
    return null
  }
  return userId
}
```

### Pitfall 3: Orphaned Clerk Users

**What goes wrong:** Clerk user created but Prisma create fails, user receives invitation but can't access app
**Why it happens:** Dual-write without rollback logic
**How to avoid:** Wrap Prisma create in try/catch, delete Clerk user on failure
**Warning signs:** Users report "invitation email received but login fails"

### Pitfall 4: Webhook Idempotency Issues

**What goes wrong:** Webhook retries create duplicate Prisma records
**Why it happens:** Clerk may retry webhooks, handler doesn't check for existing user
**How to avoid:** Use Prisma's `create()` with unique `id` field ( Clerk user ID is unique)
**Warning signs:** Prisma throws unique constraint violations (P2002 errors)

**Example:**
```typescript
// GOOD - Prisma unique constraint prevents duplicates
await prisma.user.create({
  data: {
    id: clerkUserId,  // Unique field - will throw if exists
    email: email,
    name: name,
    role: 'general_dept',
    isActive: true,
  },
})
```

### Pitfall 5: Missing Webhook Secret Configuration

**What goes wrong:** Webhook verification fails in production, all webhook calls rejected
**Why it happens:** `CLERK_WEBHOOK_SECRET` not set in production environment
**How to avoid:** Add environment variable check in webhook handler, return 500 if missing
**Warning signs:** Webhooks work locally but fail in production

**Example:**
```typescript
// GOOD - Fails fast if misconfigured
const webhookSecret = process.env.CLERK_WEBHOOK_SECRET
if (!webhookSecret) {
  return new Response('Webhook secret not configured', { status: 500 })
}
```

## Code Examples

### Verified Pattern: Complete Webhook Handler with Error Handling

```typescript
// Source: /clerk/clerk-docs - Webhooks syncing + error handling
import { verifyWebhook } from '@clerk/nextjs/webhooks'
import { NextRequest, NextResponse } from 'next/server'
import prisma from '@/lib/prisma'

export async function POST(req: NextRequest) {
  // Check environment configuration
  const webhookSecret = process.env.CLERK_WEBHOOK_SECRET
  if (!webhookSecret) {
    console.error('CLERK_WEBHOOK_SECRET not configured')
    return new NextResponse('Webhook secret not configured', { status: 500 })
  }

  try {
    // Verify webhook signature (automatic using CLERK_WEBHOOK_SECRET)
    const evt = await verifyWebhook(req)
    const eventType = evt.type

    // Handle user creation
    if (eventType === 'user.created') {
      const { id, email_addresses, first_name, last_name } = evt.data
      const email = email_addresses[0]?.email_address

      if (!email) {
        console.error('No email address in webhook payload')
        return new NextResponse('Invalid payload', { status: 400 })
      }

      // Create Prisma user record
      await prisma.user.create({
        data: {
          id: id,
          email: email,
          name: `${first_name} ${last_name}`.trim(),
          role: 'general_dept', // Default role for self-signups
          isActive: true,
        },
      })

      console.log(`✅ User created via webhook: ${id} (${email})`)
    }

    // Return success to prevent retries
    return NextResponse.json({ success: true }, { status: 200 })

  } catch (error) {
    // Log error for debugging
    console.error('❌ Webhook error:', error)

    // Check for Prisma unique constraint violation (duplicate user)
    if (error instanceof Error && error.message.includes('Unique constraint')) {
      // Idempotent: User already exists, return success
      console.log('ℹ️ User already exists (idempotent retry)')
      return NextResponse.json({ success: true, idempotent: true }, { status: 200 })
    }

    // Return 400 to prevent Clerk retries (handled via Svix)
    return NextResponse.json(
      { error: 'Webhook processing failed' },
      { status: 400 }
    )
  }
}
```

**Key Features:**
- Environment variable check (fails fast if misconfigured)
- Signature verification via `verifyWebhook()`
- Error handling for missing email addresses
- Idempotency handling (unique constraint violations treated as success)
- Proper status codes (200 for success, 400 for errors)
- Detailed logging for debugging

### Verified Pattern: Database-Backed Auth Helper

```typescript
// Source: /clerk/clerk-docs - Force token refresh guide
import { auth } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'

/**
 * Verifies the current user is authenticated and returns their userId.
 * Returns null if not authenticated.
 */
export async function requireUser() {
  const { userId } = await auth()
  if (!userId) {
    return null
  }
  return userId
}

/**
 * Verifies the current user is an admin using database-backed role check.
 * Returns null if not authenticated or not an admin.
 *
 * This approach bypasses stale JWT metadata by checking Prisma directly,
 * ensuring role changes take effect immediately.
 */
export async function requireAdmin() {
  const { userId } = await auth()
  if (!userId) {
    return null
  }

  // Check Prisma for current role (always reflects latest changes)
  const user = await prisma.user.findUnique({
    where: { id: userId },
    select: { role: true, isActive: true },
  })

  // Verify user exists, is active, and has admin role
  if (!user || !user.isActive || user.role !== 'admin') {
    return null
  }

  return userId
}
```

**Key Features:**
- Database-backed role check (real-time)
- Also checks `isActive` flag (deactivated users can't be admins)
- Returns `null` for all failure cases (consistent with Clerk patterns)
- No temporary bypasses (production-ready)

### Verified Pattern: Server Action with Transaction Safety

```typescript
// Source: /prisma/prisma - Error handling patterns
'use server'

import { auth, clerkClient } from '@clerk/nextjs/server'
import prisma from '@/lib/prisma'
import { revalidatePath } from 'next/cache'
import { requireAdmin } from '@/lib/auth'

export async function createUser(input: CreateUserInput) {
  // Require admin access
  await requireAdmin()

  let clerkUser: any

  try {
    // Step 1: Create Clerk user (source of truth)
    clerkUser = await clerkClient.users.createUser({
      emailAddress: [input.email],
      firstName: input.name.split(' ')[0],
      lastName: input.name.split(' ').slice(1).join(' '),
      publicMetadata: {
        role: input.role,
        departmentId: input.departmentId,
      },
    })

    // Note: Clerk sends invitation email automatically when no password set

  } catch (error) {
    console.error('Failed to create Clerk user:', error)
    throw new Error(`Failed to create Clerk user: ${error}`)
  }

  try {
    // Step 2: Create Prisma record
    const user = await prisma.user.create({
      data: {
        id: clerkUser.id,
        email: input.email,
        name: input.name,
        departmentId: input.departmentId,
        role: input.role,
        isActive: true,
      },
    })

    // Refresh cache
    revalidatePath('/admin/users')

    return user

  } catch (error) {
    // Step 3: Rollback Clerk user if Prisma fails
    console.error('Failed to create Prisma user, rolling back Clerk user:', error)

    try {
      await clerkClient.users.deleteUser(clerkUser.id)
      console.log('✅ Rolled back Clerk user:', clerkUser.id)
    } catch (rollbackError) {
      console.error('❌ Failed to rollback Clerk user:', rollbackError)
    }

    throw new Error(`Failed to create user record: ${error}`)
  }
}
```

**Key Features:**
- Try/catch around Clerk user creation
- Try/catch around Prisma record creation
- Rollback logic (delete Clerk user) if Prisma fails
- Detailed logging for debugging
- Proper error propagation to UI

### Verified Pattern: Middleware with Real-Time Role Checks

```typescript
// Source: /clerk/clerk-docs - Middleware protection
import { clerkMiddleware, createRouteMatcher } from '@clerk/nextjs/server'
import { NextResponse } from 'next/server'
import prisma from '@/lib/prisma'

const isAdminRoute = createRouteMatcher(['/admin(.*)'])
const isProtectedRoute = createRouteMatcher(['/dashboard(.*)', '/admin(.*)'])

export default clerkMiddleware(async (auth, req) => {
  const { userId } = await auth()

  // Protect admin routes - only users with 'admin' role can access
  if (isAdminRoute(req)) {
    if (!userId) {
      const signInUrl = new URL('/sign-in', req.url)
      return NextResponse.redirect(signInUrl)
    }

    // Database-backed role check (real-time, no cache issues)
    const user = await prisma.user.findUnique({
      where: { id: userId },
      select: { role: true },
    })

    if (!user || user.role !== 'admin') {
      console.log('❌ Redirecting non-admin user to dashboard')
      return NextResponse.redirect(new URL('/dashboard', req.url))
    }

    console.log('✅ Admin route access granted')
  }

  // Protect dashboard routes - must be authenticated
  if (isProtectedRoute(req)) {
    const { redirectToSignIn } = await auth()
    const isAuthenticated = await auth().then(a => a.isAuthenticated)
    if (!isAuthenticated) {
      return redirectToSignIn()
    }
  }
})

export const config = {
  matcher: [
    // Skip Next.js internals and all static files
    '/((?!_next|[^?]*\\.(?:html?|css|js(?!on)|jpe?g|webp|png|gif|svg|ttf|woff2?|ico|csv|docx?|xlsx?|zip|webmanifest)).*)',
    // Always run for API routes
    '/(api|trpc)(.*)',
  ],
}
```

**Key Features:**
- Database-backed role check in middleware
- Redirects non-admins to `/dashboard`
- Removes temporary bypass code
- Debug logging for monitoring

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| JWT metadata role checks | Database-backed role checks | 2025 | Role changes take effect immediately (no 60s cache delay) |

**Deprecated/outdated:**
- **`authMiddleware()`:** Replaced by `clerkMiddleware()` in Clerk v6 - use `clerkMiddleware()` from `@clerk/nextjs/server`
- **JWT metadata for authorization:** Tokens cache metadata, leads to stale role checks - use database-backed checks instead
- **Manual Svix header parsing:** Use `verifyWebhook()` utility - handles verification automatically
- **`verifyUserEmail()` for invitations:** Incorrect method - Clerk sends invitations automatically when no password set

## Open Questions

1. **Should we implement request caching for database-backed auth checks?**
   - What we know: Database queries add latency (~5-10ms per check)
   - What's unclear: Performance impact in production with high request volume
   - Recommendation: Implement without caching first, add React `cache()` if performance becomes issue

2. **Should webhook handler update Clerk metadata in addition to creating Prisma record?**
   - What we know: Current implementation updates Clerk metadata to `general_dept`
   - What's unclear: If webhook fails after Clerk update but before Prisma create, user has wrong metadata
   - Recommendation: Keep Clerk metadata update, it's idempotent and provides defense-in-depth

3. **Should we implement a dead letter queue for failed webhooks?**
   - What we know: Clerk provides retry logic via Svix, but no persistent queue
   - What's unclear: Business requirement for handling permanently failed webhooks
   - Recommendation: Add error logging, implement manual sync endpoint if needed (future phase)
