# Phase 1.2: Fix Data Consistency & User Management Issues - Context

**Gathered:** 2026-01-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Ensure data consistency between Clerk and Prisma during user creation and fix the email invitation flow. When dual-write operations fail, the system must handle rollback gracefully to prevent orphaned records. Users must receive correct invitation emails (not verification emails) and be able to complete signup.

</domain>

<decisions>
## Implementation Decisions

### Transaction rollback behavior
- If Clerk user creation succeeds but Prisma write fails, immediately call Clerk API to delete the user (auto-delete)
- This prevents orphaned Clerk accounts and maintains data consistency
- Automatic cleanup is preferred over manual cleanup despite the additional API call

### User experience during rollback failures
- If the Clerk deletion during rollback fails (leaving an orphaned user), fail silently with admin alert
- User sees generic "Something went wrong" error message
- Admin monitors for failures and handles cleanup manually

### Transaction monitoring
- Log only error scenarios (Prisma failures or rollback failures)
- Successful rollbacks are silent to reduce log noise
- Balanced approach between observability and log volume

### Error log content
- Include sanitized error context: email (sanitized), role, department ID, error message, error code
- Do NOT log full payloads or stack traces (PII concerns)
- Sufficient debugging information without exposing sensitive data

### Claude's Discretion
- Exact error message wording for user-facing errors
- Admin alert mechanism (email, console log, or dashboard notification)
- Sanitization approach for email addresses in logs (full mask, partial mask, or hash)

</decisions>
