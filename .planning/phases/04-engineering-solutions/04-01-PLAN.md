<task type="auto">
  <name>Task 1: Extend Prisma schema with Solution model and approval chain enhancements</name>
  <files>prisma/schema.prisma</files>
  <action>
    1. Add new RequestStatus values:
       - `DesignCostEstimationApproval` (engineering approval phase)
       - `FinalApproval` (final department approval phase)

    2. Create Solution model with fields:
       - id: String @id @default(cuid())
       - requestId: String (relation to Request)
       - title: String
       - description: String @db.Text
       - costEstimate: Decimal @db.Decimal(15, 2) (required, positive validation in app)
       - currency: String @default("THB")
       - timeline: String? (optional free text)
       - conceptDesign: String? @db.Text (optional description)
       - submittedById: String (relation to User)
       - submittedAt: DateTime @default(now())
       - createdAt: DateTime @default(now())
       - updatedAt: DateTime @updatedAt
       - fileAttachments: FileAttachment[] (relation for solution files)
       - Add cascade delete from Request

    2b. Extend FileAttachment model to support Solution:
       - Add optional solutionId: String? field
       - Add solution: Solution? @relation(fields: [solutionId], references: [id], onDelete: Cascade)
       - Make requestId optional (String?) since attachment can belong to either Request or Solution
       - Add constraint check: exactly one of requestId or solutionId must be set (enforce in app layer)
       - IMPORTANT: Existing file attachment queries filter by requestId and will continue to work
         since they use `where: { requestId: someId }` which matches records where requestId equals that value

    3. Add SolutionApproval model (for engineering solution approvals):
       - id: String @id @default(cuid())
       - solutionId: String (relation to Solution)
       - requiredLevel: Int? (null for custom chains)
       - requiredApproverId: String? (specific user for custom chains)
       - approverId: String? (who actually approved)
       - order: Int (sequential order 1, 2, 3...)
       - status: ApprovalStatus @default(pending)
       - comments: String? @db.Text
       - isCustomChain: Boolean @default(false)
       - approvedAt: DateTime?
       - createdAt: DateTime @default(now())
       - Add appropriate indexes on solutionId, status, order

    4. Add assignedEngineers relation to Request (many-to-many via join table RequestEngineerAssignment):
       - id: String @id @default(cuid())
       - requestId: String
       - engineerId: String
       - assignedAt: DateTime @default(now())
       - assignedById: String?
       - Indexes on requestId, engineerId

    5. Add new NotificationType values:
       - solution_ready (when solution goes to SendBackToRequester)
       - final_approval_needed (when solution enters FinalApproval)

    6. Run: npx prisma db push --accept-data-loss (development only) or npx prisma migrate dev --name add_solution_model

    IMPORTANT: Do NOT remove or modify existing RequestApproval model - it's used for department approvals. SolutionApproval is separate for engineering workflow.
  </action>
  <verify>
    - `npx prisma validate` passes without errors
    - `npx prisma db push` completes successfully
    - Solution model, SolutionApproval model, and RequestEngineerAssignment model exist in schema
    - FileAttachment model has optional solutionId field for solution attachments
    - DesignCostEstimationApproval and FinalApproval added to RequestStatus enum
    - Verify existing request file upload queries work with optional requestId:
      - Check that `prisma.fileAttachment.findMany({ where: { requestId: 'xxx' } })` returns correct results
      - Existing request-related file attachment queries should be unaffected since they filter by requestId value
  </verify>
  <done>
    Schema supports engineering solution workflow with custom approval chains, cost estimates, and engineer assignment tracking
  </done>

<task type="auto">
  <name>Task 2: Create solution server actions with custom approval chain support</name>
  <files>src/server-actions/solutions.ts</files>
  <action>
    Create new file `src/server-actions/solutions.ts` with:

    1. Zod schema for solution submission:
       ```typescript
       const submitSolutionSchema = z.object({
         requestId: z.string().cuid(),
         title: z.string().min(1).max(200),
         description: z.string().min(1).max(5000),
         costEstimate: z.number().positive('Cost must be greater than 0').min(0.01),
         currency: z.enum(['THB']).default('THB'),
         timeline: z.string().max(200).optional(),
         conceptDesign: z.string().max(2000).optional(),
         useCustomApprovals: z.boolean().default(false),
         customApproverIds: z.array(z.string()).optional(),
       })
       ```

    2. `submitSolution(input)` server action:
       - Validate user is engineering role
       - Validate request exists and status is SentToEngineer
       - Use Prisma transaction:
         a. Create Solution record
         b. Create approval chain (custom or default hierarchy)
         c. Update request status to DesignCostEstimationApproval
         d. Log activity (action: 'solution_submitted')
       - revalidatePath('/requests')
       - Return { success: true, solutionId }

    3. `createCustomApprovalChain(tx, solutionId, approverIds, submitterId)` helper:
       - Filter out submitter from approverIds (auto-skip if in chain)
       - Validate uniqueness: [...new Set(approverIds)]
       - Throw if empty after filtering
       - Create SolutionApproval records with sequential order (1, 2, 3...)
       - Set isCustomChain: true

    4. `createHierarchyApprovalChain(tx, solutionId, departmentId, submitterLevel)` helper:
       - Reuse logic from existing approvals.ts createApprovalChain
       - Create SolutionApproval records for levels above submitter
       - Set isCustomChain: false
       - Handle top-level auto-approval (create approved record immediately)

    5. `getSolutionBySolutionId(solutionId)` and `getSolutionByRequestId(requestId)` getters

    6. `getSolutionApprovals(solutionId)` - get approval chain with eligible approvers

    Use patterns from existing `src/server-actions/approvals.ts` for consistency.
  </action>
  <verify>
    - TypeScript compiles without errors: `npx tsc --noEmit`
    - File exports: submitSolution, createCustomApprovalChain, getSolutionByRequestId, getSolutionApprovals
    - Zod validation enforces positive cost estimate
  </verify>

  </done>
